## 1. 개요

Protocol 기반 DIP 로 "이벤트 수신 → 도메인 모델 변환 → 작업 큐 발행" 을 분리.

## 2. 패키지 구조

```
src/trigger_core/
├── __init__.py            # 공개 API (re-export)
├── py.typed
├── models.py              # WorkType, RepoRef, TriggerEvent
├── errors.py              # TriggerError 계층
├── protocols.py           # TriggerSource, TriggerDispatcher
├── hmac_verify.py         # verify_signature
├── dedup.py               # DeliveryCache
├── routing.py             # classify_files, DEFAULT_TASK_MAP
├── webhook.py             # WebhookTriggerSource
├── celery_dispatch.py     # CeleryTriggerDispatcher
└── mock.py                # MockTriggerEmitter
```

## 3. 인터페이스

```python
class TriggerSource(Protocol):
    def parse(self, raw: bytes, headers: Mapping[str, str]) -> TriggerEvent: ...

class TriggerDispatcher(Protocol):
    def dispatch(self, event: TriggerEvent) -> str: ...
```

`TriggerEvent.to_task_kwargs()` 로 기존 워커(comment/configaudit) 시그니처 호환.

## 4. 핵심 시퀀스

1. `WebhookTriggerSource.parse(raw, headers)` — 서명 → JSON → 스키마 → 파일 분류 → `TriggerEvent`.
2. `CeleryTriggerDispatcher.dispatch(event)` — `WorkType → (task_name, queue)` 조회 → `celery.send_task` → `task_id`.
3. `MockTriggerEmitter.emit*` — Source 우회. Dispatcher 직접 호출.

## 5. 데이터 모델

- `WorkType`: `comment` / `configaudit` (StrEnum)
- `RepoRef`: `full_name`, `clone_url`, `ref`
- `TriggerEvent`: `work_type`, `work_id`, `repo_ref`, `changed_files`, `meta`

## 6. 설정 항목 표 (그라운드 룰 §7)

본 라이브러리는 **환경변수를 직접 읽지 않는다**. 호출측 서비스(`flask-webhook`, `e2e/`) 가 자체 `Settings` 로 값을 읽어 생성자 인자로 주입한다. 따라서 모듈 단독의 설정 항목 표는 공란.

호출측 서비스가 사용하는 키(WEBHOOK_SECRET, CELERY_BROKER_URL 등) 는 각 서비스 `설계서.md` 와 `sample.env` 에서 관리.

## 7. 의존성

- `pydantic>=2.7` (models, webhook 스키마)
- `celery` **미의존** — Protocol 로 주입 (test/prod 모두 동일)

## 9. 운영 / 배포

- 이미지: `trigger-core:latest` (runtime 스테이지는 smoke test 전용). 실사용은 wheel 을 각 서비스 이미지에 포함.
- 헬스체크: 라이브러리 — 해당 없음.

## 10. SOLID

- SRP: 각 모듈 한 책임 (검증/라우팅/디스패치/도메인).
- OCP: `TriggerSource`/`TriggerDispatcher` Protocol 로 신규 구현체 추가에 기존 코드 무변경.
- LSP: `WebhookTriggerSource` / `MockTriggerEmitter` 등 구현체 교체 가능.
- ISP: Source 와 Dispatcher 분리 Protocol.
- DIP: 서비스는 Protocol 만 의존.
