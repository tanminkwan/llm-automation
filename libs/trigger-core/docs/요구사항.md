## 1. 개요

이벤트 소스(Webhook/파일/스케줄 등) 를 SCM-중립 `TriggerEvent` 로 변환하고, 작업 큐(Celery 등) 로 발행하는 경로를 **Protocol 기반 추상화** 로 분리한 공유 라이브러리.

## 2. 배경 / 목적

- P05 의 `flask-webhook` 은 HTTP 수신·HMAC 검증·Celery enqueue 가 한 서비스에 묶여 있어, 다른 이벤트 소스(파일/스케줄) 도입 시 SRP/OCP 훼손.
- E2E 에서 실제 Git/HTTP webhook 을 쓰지 않고 임의로 트리거를 발생시키려면, 수신 계층을 우회할 수 있는 동일 인터페이스가 필요.
- CLAUDE.md §1: 모듈 간 의존은 인터페이스(추상) 로만 — Dispatcher/Source 를 Protocol 로 묶으면 워커/라우트가 구현체를 모르는 상태 유지 가능.

## 3. 기능 요구사항 (FR)

| ID | 산출물 / 기능 | 설명 |
|---|---|---|
| FR-01 | `TriggerEvent`, `RepoRef`, `WorkType` | SCM-중립 도메인 모델 + 워커 kwargs 매핑 |
| FR-02 | `TriggerSource` Protocol | `parse(raw, headers) -> TriggerEvent` |
| FR-03 | `TriggerDispatcher` Protocol | `dispatch(event) -> task_id` |
| FR-04 | `WebhookTriggerSource` | HMAC 검증 + Gitea payload 파서 + `classify_files` |
| FR-05 | `CeleryTriggerDispatcher` | `celery.send_task` 래퍼, task_map 주입 가능 |
| FR-06 | `MockTriggerEmitter` | TriggerSource 우회, `TriggerEvent`/dict/JSON 3가지 입력 |
| FR-07 | `DeliveryCache` | 멱등키 TTL 캐시 |
| FR-08 | `verify_signature`, `classify_files` | 재사용 유틸 (flask-webhook 에서 이관) |
| FR-09 | 예외 계층 | `TriggerError` 최상위 + `InvalidSignatureError` / `InvalidPayloadError` / `UnsupportedWorkTypeError` / `DuplicateTriggerError` |

## 5. 제약 / 가정

- **C-01**: `TriggerSource.parse`/`TriggerDispatcher.dispatch` 는 동기 시그니처 (Q-01 결정).
- **C-02**: Gitea 이외 SCM payload 파서는 이후 Phase 에서 별도 구현체로 추가. 본 모듈은 Gitea 만 지원.
- **A-01**: Celery 는 서비스 계층에서 주입. 본 라이브러리는 celery 패키지 import 없음.
