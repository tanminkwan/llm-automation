---

## 1. 개요

Production 의 vLLM/bge-m3, Test 의 OpenAI 를 **동일한 alias 인터페이스** (`chat-llm` / `reasoning-llm` / `embedding`) 로 추상화하는 라이브러리. 호출측은 backend 구현체를 모르고, 환경변수만으로 backend 가 교체된다.

## 2. 배경 / 목적

- `architecture_test.md §1.2 T3` — backend(openai/vllm/azure) · base_url · model · api_key 는 환경변수로만 교체 가능해야 함.
- `CLAUDE.md §5.3` — 코드는 generic 이름만 참조, backend-specific 이름(`OPENAI_*` 등) 을 코드/yaml 에 직접 박지 않음.
- 다수 서비스(rag-mcp, rag-seeder, celery-*) 가 OpenAI SDK 를 직접 import 하면 backend 교체 시 N 곳을 동시 수정해야 함 → SOLID DIP 위반.
- 라이브러리화하여 wheel 로 배포 → 각 서비스 이미지 빌드 시 wheel 한 개만 끼우면 됨.

## 3. 기능 요구사항 (FR)

| ID | 산출물 / 기능 | 설명 |
|---|---|---|
| FR-01 | `ChatBackend` Protocol | `chat(messages, *, model, **kw) -> ChatResponse` 시그니처 정의 (DIP 기반) |
| FR-02 | `EmbeddingBackend` Protocol | `embed(texts, *, model) -> list[list[float]]` 시그니처 정의 |
| FR-03 | `OpenAIChatBackend` 구현체 | OpenAI 호환 SDK 호출. `base_url`/`api_key` 주입. vLLM 도 OpenAI 호환이라 동일 어댑터로 흡수 (`architecture_test.md §5.2`) |
| FR-04 | `OpenAIEmbeddingBackend` 구현체 | OpenAI Embedding API 어댑터. `dim` 메타 노출 |
| FR-05 | `LLMGateway` (alias 라우터) | `gateway.chat("chat-llm", messages=[...])` / `gateway.chat("reasoning-llm", ...)` / `gateway.embed("embedding", ...)` |
| FR-06 | `Settings` (pydantic-settings) | `architecture_test.md §12.2` 의 표준 환경변수 로딩. 직접 `os.environ` 접근 금지 |
| FR-07 | 표준 alias 3종 | `chat-llm` / `reasoning-llm` / `embedding` — 코드 상수 (production 과 동일) |
| FR-08 | 에러 모델 | `UnknownProfileError` (alias 누락), `MissingCredentialsError` (api_key 누락), `BackendCallError` (호출 실패) |
| FR-09 | 재시도 / 타임아웃 | `tenacity` 기반 exponential backoff. `*_TIMEOUT_SECONDS`, `*_MAX_RETRIES` 환경변수로 조정 (그라운드 룰 §7) |
| FR-10 | `ChatResponse` 데이터 모델 | `content: str`, `model: str`, `usage: TokenUsage`, `finish_reason: str` |

## 4. 비기능 요구사항 (NFR)

| ID | 항목 | 기준 |
|---|---|---|
| NFR-01 | 라인 커버리지 | ≥ 95% (목표 100%) |
| NFR-02 | 브랜치 커버리지 | ≥ 95% |
| NFR-03 | mypy strict | 통과 (`libs/` 전체 strict, P00 §13 Q-03) |
| NFR-04 | ruff check + format-check | 통과 |
| NFR-05 | runtime 이미지 크기 | ≤ 250MB (openai SDK 포함 — `_template` 200MB 보다 완화) |
| NFR-06 | 외부 호출 격리 | 단위 테스트는 fake/mock backend 만 사용. 실제 OpenAI 호출 0회 |
| NFR-07 | 하드코딩 금지 | 모델명/URL/타임아웃/재시도 회수 등 의미 있는 상수 0개 (그라운드 룰 §7) |
| NFR-08 | 결정성 | 가짜 backend 주입 시 호출 인자 → 응답이 결정적 (테스트 재실행 시 동일) |

## 5. 제약 / 가정

- C-01: backend 는 본 turn 에서 OpenAI 호환 1종만 (Anthropic/Azure 추가는 OCP 로 가능성만 보장).
- C-02: 동기 인터페이스만 노출 (호출측 Celery worker 가 동기 친화적). async 는 호출측이 `asyncio.to_thread` 로 래핑.
- C-03: 스트리밍(SSE) 은 본 turn 비포함 — chat/embed 단발 호출만. 스트리밍 추가는 별도 PR.
- C-04: 외부 yaml 설정 파일 없음. 환경변수만 (P01_설계서 §3.3 Q-03).
- A-01: vLLM 도 OpenAI 호환 endpoint 를 노출하므로 `OpenAIChatBackend` 가 production 에서도 그대로 동작.

## 6. 영향 범위

- **상류:** P00 워크스페이스 표준 (도구·디렉터리·Dockerfile 패턴).
- **하류:**
  - `services/rag-seeder` (P03) — `EmbeddingGateway` 사용
  - `services/rag-mcp` (P03) — `EmbeddingGateway` 사용
  - `services/celery-comment` (P05) — `LLMGateway.chat("chat-llm", ...)` 사용
  - `services/celery-configaudit` (P05) — `LLMGateway.chat("reasoning-llm", ...)` 사용
- 본 모듈의 **public API 변경** 시 위 4개 서비스 어댑터 갱신 필요. 인터페이스는 Phase 1 종료 시 고정.

## 8. 비포함 (Out of Scope)

- 스트리밍 응답(SSE) — 후속 PR 필요 시 추가.
- async 클라이언트 — 호출측 책임.
- Anthropic / Azure / Google 등 추가 backend.
- 비용/토큰 사용량 메트릭 export (Prometheus 등) — 후속 단계.
- LLM 응답 캐싱 (E2E 의 record/replay 는 P06 별도).
- yaml 외부 설정 파일 (`config/llm.yaml`).
