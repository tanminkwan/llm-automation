## 1. 개요

Config 변경 분석 워크플로우(Workflow B)에서 사용되는 MCP 서비스. `get_config_context(work_id)` 도구를 노출하여, 리발소 API(leebalso-mock)에서 dev/stage/prod 3개 환경의 http.m config 를 조회하고, 3-way diff + 이상 패턴 사전 탐지 결과를 반환한다.

## 2. 배경 / 목적

운영 config(http.m) 변경 시 dev/stage/prod 3개 환경을 수작업으로 비교하는 반복 작업을 자동화한다. `configaudit-mcp` 는 3개 환경의 before/after 를 한 번에 조회하고, 환경 간 diff 를 생성하여 LLM(reasoning-llm) 이 분석할 수 있는 구조화된 컨텍스트를 제공한다.

## 3. 기능 요구사항 (FR)

| ID | 산출물 / 기능 | 설명 |
|---|---|---|
| FR-01 | `get_config_context` MCP tool | `work_id`(str) + 선택적 `case`(str) 입력 → 3-way diff 컨텍스트 반환 |
| FR-02 | 리발소 API 조회 | dev/stage/prod 3개 환경의 `GET /config/httpm` 호출하여 before/after 수집 |
| FR-03 | 3-way diff 생성 | 각 환경의 before→after 변경을 unified diff 로 생성 |
| FR-04 | 환경 간 비교 | after 기준으로 dev↔stage, stage↔prod 환경 간 diff 생성 |
| FR-05 | 이상 패턴 탐지 | 환경 간 차이를 기반으로 이상 후보(anomaly) 리스트 생성 (예: prod 만 다른 설정) |
| FR-06 | 결과 스키마 안정화 | LLM 이 tool 결과를 파싱하므로 응답 스키마를 Pydantic 모델로 고정 |
| FR-07 | 헬스체크 | `GET /health` → `{"status": "ok"}` |

## 5. 제약 / 가정

- C-01: 리발소 API (leebalso-mock) 가 네트워크 도달 가능해야 한다 (런타임 의존).
- C-02: 환경 목록은 dev, stage, prod 3개 고정.
- A-01: 테스트 시 리발소 API 호출은 fake/mock HTTP 클라이언트로 대체.
- A-02: diff 는 Python 표준 라이브러리 `difflib` 로 생성 (외부 의존 없음).
