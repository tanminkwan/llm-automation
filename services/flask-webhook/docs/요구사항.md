## 1. 개요

Gitea push webhook 을 수신하여 HMAC 서명을 검증하고, 변경 파일의 유형(Java / http.m)에 따라 적절한 Celery 큐(`comment_queue` / `configaudit_queue`)에 태스크를 발행하는 Flask 게이트웨이 서비스.

> **Phase 6 리팩토링**: 도메인 로직(HMAC/파싱/분류/발행) 은 `libs/trigger-core` 로 이관. 본 서비스는 HTTP 어댑터로만 존재하며 `TriggerSource` / `TriggerDispatcher` Protocol 을 DI 로 수용한다 (E2E Mock 주입 가능).

## 2. 기능 요구사항 (FR)

| ID | 기능 | 설명 |
|---|---|---|
| FR-01 | Webhook 수신 | `POST /webhook/push` — Gitea push event JSON 수신 |
| FR-02 | HMAC 검증 | `X-Gitea-Signature` 헤더의 HMAC-SHA256 서명 검증. 불일치 시 401 |
| FR-03 | work_type 판별 | 변경 파일 확장자 기반: `.java` → `comment`, `.httpm` / `http.m` → `configaudit` |
| FR-04 | Celery task 발행 | `comment` → `comment_queue`, `configaudit` → `configaudit_queue` 에 `send_task` |
| FR-05 | 멱등키 | `X-Gitea-Delivery` 헤더로 중복 push 무시 (in-memory TTL 캐시) |
| FR-06 | 헬스체크 | `GET /health` → `{"status": "ok"}` |
| FR-07 | TriggerSource/Dispatcher DI | `create_app()` 은 `trigger_source`/`dispatcher`/`cache`/`celery_app` 을 주입 가능. 생략 시 `settings` 기반 기본 구현체(`WebhookTriggerSource` + `CeleryTriggerDispatcher`) 자동 구성 |
| FR-08 | public API 축약 | `flask_webhook.__all__ == {"create_app", "WebhookSettings"}` — 도메인 유틸은 `trigger_core` 에서만 import |

## 4. 제약 / 가정

- C-01: Celery broker (Redis) 가 네트워크 도달 가능해야 한다.
- C-02: Gitea webhook 은 push event 만 처리한다 (PR event 등은 미지원).
- A-01: `send_task` 로 비동기 발행만 수행. worker 모듈을 import 하지 않는다.
- A-02: 멱등키 캐시는 in-memory (프로세스 재시작 시 초기화 허용).
