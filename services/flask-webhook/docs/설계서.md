## 0. Phase 6 변경 요약

- 도메인 로직(HMAC 검증, 페이로드 파싱, work_type 분류, Celery 발행, 멱등 캐시) 은 모두 `libs/trigger-core` 로 이관.
- 본 서비스는 **HTTP 어댑터** 로만 존재 — Flask 라우트가 `TriggerSource.parse()` 와 `TriggerDispatcher.dispatch()` 를 호출할 뿐.
- `create_app()` 은 Protocol 기반 DI 를 수용 (E2E 에서 Mock 주입 가능).

## 2. 핵심 모듈

### 2.1 `app.py` — Flask 팩토리

```python
def create_app(
    *,
    trigger_source: TriggerSource | None = None,
    dispatcher: TriggerDispatcher | None = None,
    cache: DeliveryCache | None = None,
    settings: WebhookSettings | None = None,
    celery_app: Celery | None = None,
) -> Flask:
    """모든 협력자는 Protocol DI. 생략 시 settings 기반 기본 구현체 자동 구성."""
```

기본 구성:
- `trigger_source` → `WebhookTriggerSource(secret=settings.webhook_secret)`
- `dispatcher` → `CeleryTriggerDispatcher(celery_app or Celery(broker=...))`
- `cache` → `DeliveryCache(ttl_seconds=settings.dedup_ttl_seconds)`

### 2.2 `settings.py`

`WebhookSettings` (pydantic-settings) — 기존 그대로 유지. §5 참고.

## 3. API 엔드포인트

### POST /webhook/push

처리 순서 (app.py:55-82):

1. `trigger_source.parse(raw, headers)` — HMAC 검증 + 스키마 검증 + work_type 분류 일괄 위임
   - `InvalidSignatureError` → `401 {"error": "invalid signature"}`
   - `UnsupportedWorkTypeError` → `200 {"status": "skipped", "reason": "no matching files"}`
   - `InvalidPayloadError` → `400 {"error": "invalid payload"}`
2. `event.meta["delivery_id"]` 기반 멱등 확인 (HMAC 우회 방지 위해 파싱 성공 후에만)
   - `cache.is_duplicate(delivery_id)` → `200 {"status": "duplicate"}`
3. `dispatcher.dispatch(event)` → `task_id` 반환
4. `cache.mark(delivery_id)` (delivery_id 존재 시)
5. 응답: `200 {"status": "accepted", "task_id": "...", "work_type": "comment|configaudit"}`

### GET /health

- `200 {"status": "ok"}`

## 4. 설정 항목

| 키 (env) | 의미 | 기본값 | 필수 | 민감 |
|---|---|---|---|---|
| `WEBHOOK_HOST` | 바인드 호스트 | `0.0.0.0` | N | N |
| `WEBHOOK_PORT` | 리스닝 포트 | `8080` | N | N |
| `WEBHOOK_SECRET` | HMAC 시크릿 | — | Y | Y |
| `CELERY_BROKER_URL` | Redis broker URL | `redis://localhost:6379/0` | N | N |
| `DEDUP_TTL_SECONDS` | 멱등키 캐시 TTL | `300` | N | N |

> Celery task 발행 규약(task name/queue/kwargs) 과 work_type 분류 규칙은 `libs/trigger-core/docs/설계서.md` 가 단일 출처.
