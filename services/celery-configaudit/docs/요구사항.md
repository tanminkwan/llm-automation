## 1. 개요

Celery 워커 서비스. `configaudit_queue` 를 소비하여, LLM(reasoning-llm) 에이전트 루프를 실행한다. 에이전트는 ConfigAudit MCP 도구(`get_config_context`)를 tool call 로 호출하여 3-way diff 컨텍스트를 획득하고, 이를 기반으로 config 변경 분석 리포트를 생성한다.

## 2. 배경 / 목적

Workflow B(Config 변경 분석)의 실행 단위. flask-webhook 이 configaudit_queue 에 발행한 작업을 소비하고, reasoning-llm + configaudit-mcp tool call 루프를 통해 자동 분석 보고서를 생성한다.

## 3. 기능 요구사항 (FR)

| ID | 산출물 / 기능 | 설명 |
|---|---|---|
| FR-01 | Celery task `configaudit.process` | `configaudit_queue` 에서 작업 소비, AgentLoop 실행, 보고서 작성 |
| FR-02 | AgentLoop | LLM 과 MCP tool call 왕복 루프. 최대 반복 횟수 설정 가능 |
| FR-03 | ConfigAudit MCP 클라이언트 | `POST /tools/get_config_context` 호출 (httpx) |
| FR-04 | LLM 클라이언트 | OpenAI 호환 `/v1/chat/completions` 호출 + tool_call 파싱 |
| FR-05 | Markdown 리포트 생성 | AnalysisReport -> `{work_id}_{timestamp}.md` 파일 |
| FR-06 | Protocol 기반 DIP | LLMClient, ConfigAuditClient Protocol 분리 |

## 4. 비기능 요구사항 (NFR)

| ID | 항목 | 기준 (수치/조건) |
|---|---|---|
| NFR-01 | 라인 커버리지 | >= 95% |
| NFR-02 | 브랜치 커버리지 | >= 95% |

| NFR-07 | SRP 준수 | MCP 클라이언트 / LLM 클라이언트 / AgentLoop / ReportWriter / Task / Settings 분리 |

## 5. 제약 / 가정

- C-01: configaudit-mcp 서비스가 네트워크 도달 가능해야 한다 (런타임 의존).
- C-02: reasoning-llm 엔드포인트가 OpenAI 호환 API 를 제공해야 한다.
- A-01: 테스트 시 MCP, LLM 호출은 fake/mock 으로 대체.
- A-02: LLM 라이브러리를 직접 import 하지 않고 httpx 로 raw HTTP 호출.
