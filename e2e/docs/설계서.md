## 2. 테스트 계층

두 계층으로 분리:

**Offline** (기본, CI 친화적):
- `test_fixtures.py` — trigger.json 이 `TriggerEvent` 로 parse 되는지, Java fixture 트리가 존재하는지, report.schema.json 이 Draft-07 compliant 인지
- `test_emit_trigger.py` — `emit_trigger.py` CLI 가 `MockTriggerEmitter` + `CeleryTriggerDispatcher` 경로로 올바른 task_name / queue 를 `send_task` 에 전달하는지. Celery 는 fake 객체로 monkeypatch

**E2E** (`@pytest.mark.e2e`, `E2E_ENABLED=1` 미설정 시 skip):
- `test_scenario_a.py` — 실제 Celery broker 에 event 발행 → `FIXTURE_RESULT_DIR/<branch>/<ts>/` 에 snapshot 기다림 → javadoc `/**` 블록 개수 증가 확인 + `manifest.json` 검증
- `test_scenario_b.py` — 실제 Celery broker 에 event 발행 → `REPORT_OUTPUT_DIR` 에 리포트 파일 기다림 → JSON Schema 검증 + 필수 필드 non-empty

## 3. 주요 API / 흐름

### 3.1 `emit_trigger.py`

```python
celery = Celery(broker=args.broker)
dispatcher = CeleryTriggerDispatcher(celery)
emitter = MockTriggerEmitter(dispatcher)
task_id = emitter.emit_from_json(args.trigger_json)
```

exit code:
- `0` — 성공
- `2` — 파일 없음

### 3.2 `stack_up.sh` / `stack_down.sh`

- `E2E_STACK_NAME` (기본 `llmauto-e2e`) 로 격리된 stack 배포
- Redis 헬스 체크 (`redis-cli ping` → `PONG`) 대기, 타임아웃 `E2E_STACK_STARTUP_TIMEOUT`
- teardown 은 `docker stack rm` + 네트워크 정리 대기, 실패는 무시

### 3.3 `make e2e` teardown

```make
e2e:
    @set -eu; \
    trap 'bash bin/stack_down.sh' EXIT; \
    bash bin/stack_up.sh; \
    E2E_ENABLED=1 uv run --project . pytest tests
```

bash `trap ... EXIT` 으로 pytest exit code 와 무관하게 teardown 보장 (Q-05 결정).

## 4. 설정 항목

| 키 (env) | 의미 | 기본값 | 사용처 |
|---|---|---|---|
| `E2E_ENABLED` | e2e 시나리오 활성화 (`1` 이면 실행) | unset | test_scenario_* |
| `E2E_BROKER_URL` | Celery broker | `redis://localhost:6379/0` | emit_trigger, scenarios |
| `E2E_STACK_NAME` | docker stack 이름 | `llmauto-e2e` | stack_up/down |
| `E2E_STACK_STARTUP_TIMEOUT` | 헬스 대기 초 | `120` | stack_up |
| `E2E_TASK_TIMEOUT` | 워커 완료 대기 초 | `180` | scenarios |
| `FIXTURE_RESULT_DIR` | celery-comment snapshot 출력 루트 | `/fixtures/results` | test_scenario_a |
| `REPORT_OUTPUT_DIR` | celery-configaudit 리포트 루트 | `/reports` | test_scenario_b |

## 5. 시나리오 A 구조적 검증 기준

| 항목 | 기준 |
|---|---|
| snapshot 존재 | `{FIXTURE_RESULT_DIR}/main/<ts>/` 발견 |
| 파일 존재 | `Calculator.java` 가 snapshot 내에 존재 |
| 주석 증가 | `/**` 블록 개수가 `before` 대비 증가 (Q-04: 문자열 diff 아닌 카운트) |
| manifest | `manifest.json` 의 `branch`, `message` 에 work_id 포함 |

## 6. 시나리오 B 구조적 검증 기준

| 항목 | 기준 |
|---|---|
| 리포트 발견 | `{REPORT_OUTPUT_DIR}/**/*.{md,json}` 중 work_id 일치 파일 |
| 스키마 통과 | Draft-07 validator (`work_id`, `case`, `summary`, `anomalies`, `details`, `generated_at`) |
| non-empty | `summary` 문자열이 공백 아님, `anomalies` 는 list |

LLM 자연어 내용은 **존재 + 비어있지 않음** 만 검증 — 자연어 변동에 과민반응하지 않음.
