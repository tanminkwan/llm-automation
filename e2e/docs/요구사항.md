## 1. 개요

Phase 6 의 E2E 검증 Harness. 실제 Gitea/git I/O 를 사용하지 않고 **사전 준비된 픽스처 파일** 로 워크플로우 A(Java 주석) 와 B(ConfigAudit) 를 재현한다.

- 트리거: `trigger_core.MockTriggerEmitter` 로 `TriggerEvent` 를 Celery 큐에 직접 발행
- Git: `FixtureGitClient` 가 `fixture://<key>` URL 을 파일 복사로 해석
- MCP: rag-mcp / configaudit-mcp 는 **실제 컨테이너** 띄워 HTTP 호출
- LLM: OpenAI API 직접 호출 (test alias — chat-llm / reasoning-llm)

## 2. 기능 요구사항 (FR)

| ID | 기능 | 설명 |
|---|---|---|
| E2E-FR-01 | trigger.json 스키마 | `TriggerEvent` 로 parseable 한 JSON 파일 제공 (시나리오별) |
| E2E-FR-02 | fixture 소스 트리 | 시나리오 A: Java 소스 트리. `fixture://scenario_A/before` 로 clone 됨 |
| E2E-FR-03 | CLI 발행 | `e2e/bin/emit_trigger.py <trigger.json>` → Celery 큐에 발행 |
| E2E-FR-04 | stack 기동/정리 | `bin/stack_up.sh`, `bin/stack_down.sh` — docker stack deploy/rm + 헬스 대기 |
| E2E-FR-05 | 시나리오 A 검증 | 워커 결과 snapshot 에서 Java javadoc(`/**`) 블록 수 증가 확인 |
| E2E-FR-06 | 시나리오 B 검증 | 워커 결과 리포트 파일 생성 + JSON Schema draft-07 + 필수 필드 non-empty |
| E2E-FR-07 | offline 검증 | fixture/CLI 구조적 유효성은 stack 없이도 항상 실행 (오프라인 단위 커버리지) |
| E2E-FR-08 | teardown 보장 | `make e2e` 는 pytest 결과와 무관하게 `bash` trap 으로 `stack_down.sh` 실행 |

## 4. 제약 / 가정

- C-01: 실제 Gitea clone/push 를 사용하지 않는다 (memory: feedback_e2e_no_git_io).
- C-02: OpenAI API 호출은 실행 시마다 비용 발생. `E2E_ENABLED=1` 미설정이면 skip.
- A-01: docker Swarm 은 단일 노드 모드로 가정 (`docker stack deploy`).
- A-02: `FIXTURE_RESULT_DIR`, `REPORT_OUTPUT_DIR` 는 호스트 ↔ 컨테이너 볼륨으로 공유.
