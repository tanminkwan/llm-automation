## 1. 개요

Production 과 동일한 **alias 인터페이스** (`chat-llm` / `reasoning-llm` / `embedding`) 를 제공하는 LLM/Embedding 추상화 계층 (`libs/llm-gateway`) 과, OpenCode 등 외부 AI Agent CLI 를 동일 시그니처로 호출하기 위한 추상화 계층 (`libs/agent-runner`) 을 확립한다. 후속 모든 서비스(rag-mcp, configaudit-mcp, celery-comment, celery-configaudit) 는 본 라이브러리만 의존하며, backend 교체는 환경변수만으로 가능해야 한다.

## 2. 배경 / 목적

- `architecture_test.md §1.2 T3/T6` 는 LLM/Agent 백엔드를 환경변수로만 교체 가능해야 함을 명시.
- production(vLLM) ↔ test(OpenAI) 가 **alias·env 키·manifest** 를 공유하고 *값* 만 다름 (`CLAUDE.md §5`).
- 다수 서비스가 동일한 OpenAI/MCP 어댑터를 직접 import 하면 컨테이너 독립성(그라운드 룰 §1) 위반.
- 추상화를 라이브러리로 분리해 wheel 로 배포 → 각 서비스 이미지에는 wheel 만 포함.

## 3. 기능 요구사항 (산출물)

| ID | 산출물 | 설명 |
|---|---|---|
| FR-01 | `libs/llm-gateway` 라이브러리 | `ChatBackend` / `EmbeddingBackend` Protocol + OpenAI 호환 어댑터 + alias 로더 |
| FR-02 | `libs/agent-runner` 라이브러리 | `AgentRunner` Protocol + `OpenCodeRunner` subprocess 어댑터 + 결과 모델 |
| FR-03 | 표준 alias 3종 | `chat-llm` / `reasoning-llm` / `embedding` — production 과 동일 (`CLAUDE.md §5.1`) |
| FR-04 | 환경변수 표준 | `*_BASE_URL` / `*_MODEL` / `*_API_KEY` (+ `EMBEDDING_DIM`) — `architecture_test.md §12.2` 단일 출처 |
| FR-05 | wheel 빌드 | 두 라이브러리 모두 `make build` 로 배포 가능한 wheel 생성 |
| FR-06 | 검증용 컨테이너 이미지 | 각 라이브러리의 4-stage Dockerfile (P00 표준 준수) — runtime 은 단순 import 스모크 |
| FR-07 | 본 phase 의 테스트결과서 | `docs/P01_테스트결과서.md` — 두 모듈 커버리지/이미지/DoD 종합 |

> **본 턴 범위:** FR-01 (llm-gateway) end-to-end. FR-02 (agent-runner) 는 후속 턴에 동일 절차로 진행 후 FR-07 로 종합.

## 5. 제약 / 가정

- C-01: 도구 스택은 P00 확정안 (Python 3.12, uv, pytest, ruff, mypy, multi-stage Dockerfile).
- C-02: vLLM 도 OpenAI 호환 API 를 노출하므로 단일 OpenAI 어댑터로 흡수 (`architecture_test.md §5.2`).
- C-03: 본 Phase 는 인프라(redis/qdrant/gitea) 미기동 — 라이브러리 단위 테스트만.
- C-04: `libs/agent-runner` 의 OpenCode 결합 방식(host 호출 vs sidecar) 결정은 P05 로 이월 (`구현계획.md §11`). 본 라이브러리는 결합 방식과 무관한 인터페이스만 제공.
- A-01: production/test 가 동일 alias 사용 — 모듈 코드는 backend 식별자(`openai`, `vllm`) 를 직접 분기하지 않음.

## 6. 영향 범위

- **상류:** P00 의 워크스페이스/도구 표준.
- **하류:**
  - `services/rag-seeder`, `services/rag-mcp` → `EmbeddingGateway` 사용 (P03)
  - `services/configaudit-mcp` → 직접 의존은 없으나 결과 schema 가 LLM 입력에 영향 (P04)
  - `services/celery-comment`, `services/celery-configaudit` → `LLMGateway` + `AgentRunner` 모두 사용 (P05)
- 본 라이브러리 인터페이스 변경 시 위 5개 서비스의 어댑터 갱신 필요 → 인터페이스는 P01 종료 시 **확정**, 변경은 PR + 본 문서 갱신.
