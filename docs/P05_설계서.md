## 1. 아키텍처 개요

```
Gitea push
  ↓
flask-webhook (HMAC verify → classify → dispatch)
  ├── comment_queue ──→ celery-comment (clone → prompt → agent → commit)
  └── configaudit_queue ──→ celery-configaudit (agent loop → MCP → report)
```

세 서비스는 **Celery + Redis** 로 느슨 결합된다. flask-webhook 은 `send_task()` 로 태스크를 발행하고, celery worker 들이 각 큐를 독립 소비한다.

## 2. 모듈별 설계 요약

### 2.1 flask-webhook

| 계층 | 모듈 | 책임 |
|---|---|---|
| Entry | `app.py` | Flask factory (`create_app`) |
| Verification | `hmac_verify.py` | `verify_signature()` — HMAC-SHA256 |
| Dedup | `dedup.py` | `DeliveryCache` — TTL 기반 in-memory 캐시 |
| Routing | `router.py` | `classify_files()` → `WorkType` enum, `dispatch()` → Celery `send_task` |
| Models | `models.py` | `WebhookPayload`, `PushCommit` (Pydantic) |
| Config | `settings.py` | `WebhookSettings` (pydantic-settings) |

- **DIP**: `create_app(celery_app, cache)` — 의존성 주입으로 테스트 시 fake 교체 가능
- **Dispatch 규칙**: `comment.process` → `comment_queue`, `configaudit.process` → `configaudit_queue`
- **work_id**: `{repo_full_name}:{after_sha[:8]}`

### 2.2 celery-comment

| 계층 | 모듈 | 책임 |
|---|---|---|
| Task | `task.py` | Celery task `comment.process` |
| Core | `task.py:process_comment()` | 오케스트레이션 (clone → prompt → agent → commit) |
| Git | `git_ops.py` | `SubprocessGitClient` — clone, add_commit_push |
| Prompt | `prompt.py` | `build_prompt()` — RAG 컨텍스트 기반 프롬프트 생성 |
| Protocols | `protocols.py` | `GitClient`, `AgentExecutor`, `AgentExecResult` |
| Config | `settings.py` | `CeleryCommentSettings` |

- **DIP**: `process_comment(payload, git_client, agent, ...)` — Protocol 기반 인터페이스
- Git 작업은 subprocess 호출, 테스트 시 fake 구현

### 2.3 celery-configaudit

| 계층 | 모듈 | 책임 |
|---|---|---|
| Task | `task.py` | Celery task `configaudit.process` |
| Core | `agent_loop.py` | `AgentLoop` — LLM ↔ tool_call 반복 |
| LLM | `llm_client.py` | `OpenAILLMClient` — OpenAI 호환 API 호출 |
| MCP | `mcp_client.py` | `HttpConfigAuditClient` — configaudit-mcp POST |
| Report | `report_writer.py` | `write_report()` — Markdown 보고서 생성 |
| Models | `models.py` | `TaskPayload`, `ToolCallRequest`, `AnalysisReport` |
| Protocols | `protocols.py` | `ConfigAuditClient`, `LLMClient` |
| Config | `settings.py` | `CeleryConfigAuditSettings` |

- **Agent Loop 패턴**: LLM 응답에 `tool_calls` 가 있으면 MCP 호출 → 결과를 메시지에 추가 → 재호출, 없으면 최종 텍스트로 보고서 생성
- `max_iterations` 로 무한 루프 방지
- 보고서 파싱: JSON 시도 → 실패 시 텍스트 폴백

| 환경변수 | 의미 | 기본값 | 필수 |
|---|---|---|---|
| `WEBHOOK_HOST` | 바인드 호스트 | `0.0.0.0` | N |

## 4. 테스트 전략

- 각 서비스 독립 단위 테스트 (fake/mock 사용)
- LLM/MCP/Git 외부 의존성은 모두 Protocol + fake 구현으로 격리
- 커버리지 ≥ 95% (브랜치 포함)
- E2E 통합 테스트는 Phase 6 에서 수행
