## 1. 개요

Phase 6 은 Phase 1~5 에서 만든 모든 서비스를 단일 호스트 docker stack 위에서 구동하고, 두 개의 핵심 워크플로우(Java 주석 추가 / Config 감사)를 **트리거 발생 → 큐 → 워커 → MCP 도구 → 결과 파일** 의 엔드-투-엔드 경로로 검증한다.

동시에 `flask-webhook` 에 암묵적으로 고착돼 있던 "HTTP webhook 수신 = 작업 트리거" 가정을 `TriggerSource` 추상화 계층으로 분리해, 다른 이벤트 소스(파일/스케줄/외부 큐)로 확장 가능한 구조로 재편한다.

## 2. 배경 / 목적

1. **통합 검증 부재 해소** — Phase 1~5 는 모듈 단위 테스트만 확보. 서비스 간 실제 HTTP/큐 통신을 재현하는 경로가 없음.
2. **Trigger 결합도 제거** — 현재 `services/flask-webhook` 은 Gitea webhook 수신 로직·HMAC 검증·Celery enqueue 가 한 서비스에 묶여 있어, 운영상 다른 이벤트 소스 도입 시 수정이 크다. OCP 위반.
3. **재현 가능한 테스트** — 실제 Git 서버(Gitea) + 네트워크 + SCM 인증을 E2E 에 끌어들이면 결정성이 깨진다. 사전 준비한 fixture 파일로 입력·기대결과를 고정한다.
4. **CI 플랫폼 비의존** — 운영 환경은 GitHub Actions 등 외부 CI 에 결속되지 않는다. 본 Phase 는 `make e2e` 한 방으로 완결.

## 3. 기능 요구사항 (FR)

### 3.1 `libs/trigger-core` (신규 공유 라이브러리)

| ID | 기능 | 설명 |
|---|---|---|
| TC-FR-01 | `TriggerSource` Protocol | `receive() -> TriggerEvent` 추상 메서드 (또는 동기/비동기 동등 시그니처). 구현체 교체 가능 |
| TC-FR-02 | `TriggerEvent` 도메인 모델 | `work_type`, `work_id`, `repo_ref`, `changed_files`, `meta` (Pydantic). SCM 중립 |
| TC-FR-03 | `WebhookTriggerSource` 구현 | HTTP raw 요청 + HMAC 시크릿 → 검증 후 `TriggerEvent` 로 변환. Gitea payload 파서 포함 |
| TC-FR-04 | `MockTriggerEmitter` 구현 | 테스트/로컬 용. `TriggerEvent` 를 직접 주입 (파일 경로, dict, 또는 파이썬 호출). 실 네트워크 불필요 |
| TC-FR-05 | `TriggerDispatcher` | `TriggerEvent` → Celery 태스크 enqueue. 태스크 라우팅 규칙(work_type→queue) 주입 가능 |
| TC-FR-06 | 에러 타입 | `InvalidSignatureError`, `UnsupportedWorkTypeError`, `DuplicateTriggerError` |

### 3.2 `services/flask-webhook` 리팩토링

| ID | 기능 | 설명 |
|---|---|---|
| FW2-FR-01 | HTTP 어댑터화 | Flask 라우트는 **얇은 어댑터** 로만 남기고, 검증/파싱은 `WebhookTriggerSource` 에 위임 |
| FW2-FR-02 | Dispatcher 주입 | `TriggerDispatcher` 를 DI 로 받아 enqueue. Celery 직접 import 제거 |
| FW2-FR-03 | 기존 엔드포인트/스펙 유지 | `POST /webhook/push`, `GET /health` 응답 바디·상태 코드 동일 |
| FW2-FR-04 | 기존 단위 테스트 통과 | Phase 5 테스트는 리팩토링 후에도 그대로 통과 |

### 3.3 `services/celery-comment` — `FixtureGitClient` 추가

| ID | 기능 | 설명 |
|---|---|---|
| CCE-FR-01 | `FixtureGitClient` 구현 | 기존 `GitClient` Protocol 의 신규 구현체. `clone()` = 지정 디렉터리를 work_dir 로 복사. `add_commit_push()` = 결과 디렉터리로 스냅샷 기록 |
| CCE-FR-02 | 구현체 선택 규칙 | 환경변수(예: `GIT_CLIENT=subprocess\|fixture`) 로 런타임 전환. 기본은 `subprocess` (prod 유사) |
| CCE-FR-03 | 기존 경로 무영향 | `SubprocessGitClient` 경로는 기존 동작·커버리지 유지 |

### 3.4 `e2e/` (신규 디렉터리)

| ID | 기능 | 설명 |
|---|---|---|
| E2E-FR-01 | 시나리오 A — Java 주석 | `MockTriggerEmitter` → `comment_queue` → `celery-comment`(FixtureGitClient + RAG MCP + `chat-llm`) → 결과 파일이 `expected/` 와 의미적으로 동등한 주석 포함 |
| E2E-FR-02 | 시나리오 B — Config 감사 | `MockTriggerEmitter` → `configaudit_queue` → `celery-configaudit`(ConfigAudit MCP tool call + `reasoning-llm`) → `AnalysisReport` Markdown 파일 생성 및 필수 섹션 존재 |
| E2E-FR-03 | 픽스처 배치 | `e2e/fixtures/scenario_A/{before,trigger.json,expected}/`, `e2e/fixtures/scenario_B/<case>/{trigger.json,expected}/` |
| E2E-FR-04 | 스택 부팅/종료 | `make e2e` 가 `docker stack deploy` (test stack) → pytest 실행 → teardown 까지 자동 수행 |
| E2E-FR-05 | MCP 실제 호출 | `rag-mcp`, `configaudit-mcp`, `leebalso-mock` 는 **실제 컨테이너로 기동**, 워커가 HTTP 로 호출 (in-process fake 불가) |
| E2E-FR-06 | LLM 실제 호출 | `chat-llm`, `reasoning-llm` 은 **OpenAI 직접 호출**. `sample.env` 의 API 키를 stack 에 주입. record/replay 미도입 |
| E2E-FR-07 | Mock Trigger 스크립트 | `e2e/bin/emit_trigger.py`(또는 pytest fixture) 로 시나리오별 payload 를 큐에 직접 발행. HTTP 서버 불필요 |
| E2E-FR-08 | 결과 검증 | `expected/` 와의 diff 비교 (주석 시나리오) + 리포트 스키마 필수 필드 존재 검사 (감사 시나리오). LLM 출력 변동은 **의미적 검증**(키 존재·섹션 구조) 으로 완화 |

## 5. 제약 / 가정

- **C-01**: 본 Phase 는 실제 Gitea push 이벤트를 재현하지 않는다. Webhook 수신 경로는 `libs/trigger-core` 단위 테스트로만 검증.
- **C-02**: `celery-comment` 의 `FixtureGitClient` 는 **E2E 전용**. Prod/테스트-prod-유사 경로에서는 `SubprocessGitClient` 를 사용한다.
- **C-03**: LLM 은 OpenAI 실호출. 오프라인 실행 시 E2E 는 실패한다 (의도된 제약).
- **A-01**: docker swarm (single-node) 이 호스트에 init 되어 있다 (`infra/scripts/init-swarm.sh` 선행).
- **A-02**: `OPENAI_API_KEY` 가 `.env` 에 존재한다. 누락 시 `make e2e` 는 선행 체크에서 실패.
- **A-03**: E2E 시나리오에서 호출되는 LLM 응답은 **구조적 요구사항**(예: tool_call 포맷, 리포트 섹션 존재)을 만족하도록 프롬프트가 설계돼 있다 (Phase 5 에서 확보).
