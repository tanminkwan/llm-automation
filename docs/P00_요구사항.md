---

## 1. 개요

후속 모든 Phase 가 동일한 도구·디렉터리·문서·CI 표준을 따를 수 있도록 **프로젝트의 골격(skeleton) 과 표준(standard) 을 확립** 한다. Phase 0 가 종료되면, 이후 어떤 모듈을 추가하더라도 **동일한 절차(`make test`, `make coverage`, `make image`)** 로 빌드·테스트·이미지화가 가능해야 한다.

## 2. 배경 / 목적

- 프로젝트는 **컨테이너 단위 독립 release** 와 **TDD/커버리지 95%** 를 그라운드 룰로 한다(`CLAUDE.md §1`).
- 이를 매 모듈마다 즉흥적으로 세팅하면 일관성·재현성이 깨진다.
- Phase 0 에서 표준을 한 번 굳히고, 이후 Phase 는 **표준을 복사·확장만** 한다.

## 4. 비기능 요구사항 (품질 기준)

| ID | 항목 | 기준 |
|---|---|---|
| NFR-01 | 재현성 | clean clone 직후, `cp sample.env .env && cd libs/_template && make coverage` 한 번에 통과 |
| NFR-02 | 커버리지 | `_template` 라이브러리 라인+브랜치 커버리지 ≥ 95% |
| NFR-03 | 린트/타입체크 | `make lint` 가 ruff(check+format-check) + mypy(strict) 모두 통과 |
| NFR-04 | 이미지 크기 | `_template` runtime 이미지 ≤ 200MB (multi-stage 효과 검증) |
| NFR-05 | 표준 문서 가독성 | 템플릿 3종이 빈 모듈 1개에 적용되어 실제 사용 가능 (lorem ipsum 아님) |
| NFR-06 | 환경변수 동기화 | `sample.env` 의 키 ↔ `pyproject.toml`/`설계서.md "설정 항목"` 표 ↔ stack.yml 매핑이 1:1 일치 (Phase 0 한정) |

## 5. 제약 / 가정

- C-01: 도구 스택은 `CLAUDE.md §4` 확정안 사용 (Python 3.12, uv, pytest, ruff, mypy, multi-stage Dockerfile).
- C-02: CI 는 옵션 B (Makefile 우선, Actions yaml 미작성). `Makefile` 만으로 그라운드 룰 검증 가능해야 함.
- C-03: 본 Phase 에서는 외부 서비스(OpenAI, Qdrant 등) 호출 없음 — `_template` 은 순수 in-process 로직만.
- C-04: 단일 호스트 Linux 환경 (개발자 머신) 가정.

## 6. 영향 범위

- **상류**: 없음 (시작 phase).
- **하류**: 모든 후속 phase 의 모듈 디렉터리 구조·도구·CI·문서 절차가 본 Phase 산출물에 의존.
- 본 Phase 가 정한 표준 변경 시 후속 모든 모듈에 전파 비용 발생 → 변경은 **PR + 본 문서 갱신** 으로만.

## 7. 완료 기준 (Definition of Done)

다음을 **모두 충족** 해야 P00 종료 → P01 진입 가능.

- [ ] FR-01 ~ FR-10 산출물 모두 존재
- [ ] `cd libs/_template && make lint coverage image` 가 clean 상태에서 통과
- [ ] `_template` 라인+브랜치 커버리지 ≥ 95% (수치를 `P00_테스트결과서.md` 에 기록)
- [ ] 런타임 이미지 크기 ≤ 200MB
- [ ] `sample.env` ↔ `pyproject.toml` ↔ 설계서 설정 항목 표 키 일치
- [ ] `docs/P00_테스트결과서.md` 작성 완료 (실행 명령/통과 수/커버리지 수치/이미지 크기 포함)
- [ ] 본 문서(P00_요구사항.md), `P00_설계서.md`, `P00_테스트결과서.md` 3종 모두 commit

## 8. 비포함 (Out of Scope)
