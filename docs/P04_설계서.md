## 1. 개요

Phase 4 는 단일 모듈(`services/configaudit-mcp`)로 구성된다. 아키텍처·인터페이스 상세는 모듈 설계서를 참조한다.

- **모듈 설계서**: `services/configaudit-mcp/docs/설계서.md`

## 2. 아키텍처 요약

```
AI Agent
    ↓ POST /tools/get_config_context(work_id, case)
FastAPI App (app.py, port 9002)
    ↓ Analyzer.analyze(work_id, case)
    ├→ ConfigFetcher × 3 (dev/stage/prod)
    │   ↓ httpx GET → leebalso-mock:9100
    ├→ DiffEngine.diff_before_after() × 3
    ├→ DiffEngine.diff_cross_env() × 2
    └→ AnomalyDetector.detect()
    ← AuditContext (JSON)
```

## 3. 핵심 설계 결정

| 결정 | 근거 |
|---|---|
| `ConfigClient` Protocol (DIP) | 테스트 시 FakeConfigClient 주입으로 HTTP 의존 제거 |
| `create_app()` 팩토리 패턴 | 테스트 시 Analyzer 주입, uvicorn `--factory` 모드 지원 |
| `difflib.unified_diff` | 외부 의존 없음, LLM 파싱에 보편적인 형식 |
| 이상 탐지 규칙: 1-env-differs | 3환경 중 1개만 다르면 anomaly. 단순 규칙으로 시작, 확장 가능 |
| Pydantic BaseModel 응답 | 스키마 안정화 + FastAPI 자동 직렬화 |

## 4. SOLID 적용

| 원칙 | 적용 |
|---|---|
| SRP | ConfigFetcher / DiffEngine / AnomalyDetector / Analyzer / App / Settings — 각 단일 책임 |
| OCP | 탐지 규칙 추가 = AnomalyDetector 확장, diff 알고리즘 교체 = DiffEngine 만 수정 |
| LSP | ConfigClient Protocol 구현체는 모두 교체 가능 |
| ISP | App → Analyzer.analyze() 만 호출 |
| DIP | Analyzer 는 ConfigClient Protocol 에만 의존 |

## 5. 설정 항목

| 키 (env) | 의미 | 기본값 | 필수 | 민감 |
|---|---|---|---|---|
| `CONFIGAUDIT_HOST` | 바인드 호스트 | `0.0.0.0` | N | N |
| `CONFIGAUDIT_PORT` | 리스닝 포트 | `9002` | N | N |
| `CONFIGAUDIT_DEFAULT_CASE` | 기본 case ID | `case-001` | N | N |
| `LEEBALSO_BASE_URL` | 리발소 API base URL | `http://localhost:9100` | N | N |
| `LEEBALSO_TIMEOUT` | 리발소 API 타임아웃(초) | `10.0` | N | N |
